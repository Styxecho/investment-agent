## 1. 文件说明

本文档详细说明 `etf_universe.csv` 中每个字段的定义、数据来源、清洗规则和维护注意事项。

### 文件演进
- `v1`：原始手工整理版本，表头为中文，存在换行符和格式不统一问题
- `v2`：经脚本清洗后的标准化版本，表头改为英文简写，数据格式化，并新增 `pool_role`（组合角色）一列。`market_cap_label`（市值标签）经扩展后同时承担市值/风格/久期定位功能

## 2. 字段说明

| 字段名 | 中文含义 | 数据类型 | 说明 | 示例 |
|--------|---------|---------|------|------|
| `code` | 证券代码 | 字符串 | ETF 在交易所的交易代码，含后缀（.SH / .SZ） | `510300.SH` |
| `name` | 证券名称 | 字符串 | ETF 的简称 | `沪深300ETF华泰柏瑞` |
| `index_code` | 跟踪指数代码 | 字符串 | 同花顺指数代码，部分商品/债券 ETF 可能为空 | `000300.SH` |
| `index_name` | 跟踪指数名称 | 字符串 | 指数全称 | `沪深300指数` |
| `fund_size` | 基金规模 | 数值（亿元） | 最新基金规模。部分货基/债券 ETF 在原数据中缺失。已去除千分位逗号 | `2062.3743` |
| `tracking_error` | 年跟踪误差 | 数值（%） | 相对跟踪指数的年化跟踪误差。部分 ETF 数据缺失（空值表示） | `0.0230` |
| `daily_turnover` | 年日均成交量 | 数值（亿元） | 年日均成交额，衡量流动性。对于货基，此字段可能比 fund_size 更能反映实际规模 | `45.1593` |
| `mgmt_fee` | 管理费率 | 数值（%） | 年化管理费率 | `0.1500` |
| `custody_fee` | 托管费率 | 数值（%） | 年化托管费率 | `0.0500` |
| `total_fee` | 总费率 | 数值（%） | 管理费率 + 托管费率，综合费率口径 | `0.2000` |
| `asset_class_l1` | 大类资产_L1 | 字符串 | 一级资产分类 | `股票` / `债券` / `商品` / `货基` |
| `asset_class_l2` | 大类资产_L2 | 字符串 | 二级资产分类 | `宽基` / `策略风格` / `行业主题` / `港股宽基` / `海外宽基` / `中期国债` / `黄金` 等 |
| `sector_l1` | 行业主题_L1 | 字符串 | 一级行业或主题（若适用） | `科技` / `医药` / `消费` / `制造` / `周期` / `金融地产` |
| `sector_l2` | 行业主题_L2 | 字符串 | 二级细分行业或主题（若适用） | `芯片` / `人工智能` / `生物医药` / `新能源` / `银行` / `证券` 等 |
| `market_cap_label` | 市值标签 | 字符串 | **v2 扩展**。原数据中的市值定位，经扩展后同时承担市值/风格/久期定位功能（如大盘/中盘/科创大盘/港股大盘/短久期利率/黄金/纳斯达克等），用于核心池去重和分类 | `大盘` / `中盘` / `中小盘` / `小盘` / `科创大盘` / `港股大盘` / `短久期利率` / `黄金` / `纳斯达克` 等 |
| `strategy_style` | 策略风格 | 字符串 | Smart Beta 或策略因子标签（若适用） | `红利` / `低波` / `价值` / `成长` / `质量` / `自由现金流` / `央国企` |
| `pool_role` | 组合角色 | 字符串 | **v2 新增**。该 ETF 在核心-卫星框架中的角色 | `core` / `satellite` / `extended` |

## 3. 关键字段详解

### 3.1 `market_cap_label`（市值标签，v2 扩展用途）

在 v2 版本中，`market_cap_label` 不再仅仅是原数据中的"大盘/中盘/小盘"标签，而是被**扩展为市值/风格/久期/市场的综合定位字段**，用于解决同一大类资产下多只 ETF 的区分问题，以及核心池的去重和筛选。

**设计原则**：
- 对于**股票宽基**：描述其在市值谱系中的精确位置（大盘/中盘/小盘/科创大盘等）
- 对于**策略风格 ETF**：描述其市值定位（大盘/中盘/小盘/全市场），策略因子由 `strategy_style` 单独承担
- 对于**债券 ETF**：按久期和信用属性划分（短久期利率/中久期利率/长久期利率/信用债/可转债）
- 对于**商品/跨境 ETF**：按具体品种或市场划分（黄金/纳斯达克/标普500/日本股市/港股大盘）
- 对于**行业主题 ETF**：暂时为空，未来可补充市值定位

**主要取值及含义**：

| 取值 | 适用对象 | 含义 |
|------|---------|------|
| `大盘` | A股宽基 | 沪深300、上证50、A500、深证100等大盘股指数 |
| `中盘` | A股宽基 | 中证500为代表的中盘股指数 |
| `中小盘` | A股宽基 | 中证1000为代表的中小盘股指数 |
| `小盘` | A股宽基 | 中证2000为代表的小盘股指数 |
| `科创大盘` | 科创板宽基 | 科创50（科创板规模最大50只） |
| `科创中盘` | 科创板宽基 | 科创100（科创板中盘100只） |
| `科创小盘` | 科创板宽基 | 科创200（科创板小盘200只） |
| `科创全市场` | 科创板宽基 | 科创综指（科创板全部股票） |
| `港股大盘` | 港股宽基 | 恒生指数、恒生国企指数等 |
| `纳斯达克` | 海外宽基 | 纳斯达克100指数 |
| `标普500` | 海外宽基 | 标准普尔500指数 |
| `日本股市` | 海外宽基 | 日经225、东证指数 |
| `全市场` | 策略风格 | 选股范围覆盖全市场，不区分市值（如红利ETF覆盖全市场） |
| `短久期利率` | 债券 | 0-3年、0-4年国债/政金债 |
| `中久期利率` | 债券 | 5年、5-10年国债/政金债 |
| `长久期利率` | 债券 | 10年、30年国债/政金债 |
| `短久期信用` | 债券 | 短融、1年内信用债 |
| `信用债` | 债券 | 公司债、城投债、基准做市信用债 |
| `可转债` | 债券 | 可转债及可交换债指数 |
| `黄金` | 商品 | 黄金现货合约 |
| `货币基金` | 货基 | 场内货币市场基金 |

### 3.2 `pool_role`（组合角色）

该字段标记每只 ETF 在"核心-卫星"框架中的定位。

| 取值 | 含义 | 数量级（当前v2） |
|------|------|----------------|
| `core` | **核心池**。长期底仓，规模大、流动性好、费率低、代表性强 | 约 10-15 只 |
| `satellite` | **卫星池**。用于主动配置、行业轮动、策略增强 | 约 50-100 只 |
| `extended` | **扩展池**。通过初筛但暂不纳入常规研究，作为观察或后备 | 其余 |

**v2 版本的自动标记规则**（需人工审核）：
- **核心池**：在每个 `asset_class_l2` + `market_cap_label` 组合下，选择**规模最大的一只**
- **卫星池**：未入选核心池，但标签清晰、流动性合格的行业主题和策略风格 ETF
- **扩展池**：通过初筛但同质性高、标签不明确、或规模明显落后的标的

**特别说明**：
- `core` 的自动标记基于"同类规模最大"规则，**必须经人工审核确认**
- 用户可以根据自身偏好调整核心池（例如将科创100从 satellite 提升为 core）
- 货币基金的 `fund_size` 在原数据中部分缺失，标记结果可能不准确，需优先参考 `daily_turnover`

## 4. 数据清洗规则（v2 生成时执行）

1. **表头标准化**：将包含换行符的复杂中文表头替换为英文简写
2. **数值清洗**：
   - 去除 `fund_size` 中的千分位逗号（如 `2,062.3743` → `2062.3743`）
   - 将 `--`、`-` 等缺失标记替换为空字符串
   - 去除数值字段首尾空格
3. **字段新增**：
   - `pool_role`：基于 `asset_class_l2` + `market_cap_label` 分组规模和角色规则自动标记

## 5. 维护规范

### 5.1 更新频率
- **Review 周期**：初期每年一次，逐步提高频率，不高于季频
- **触发更新的场景**：
  - 新增 ETF 产品需要纳入备选池
  - 某只 ETF 规模跌破 2 亿或流动性显著恶化
  - 跟踪误差异常扩大
  - 费率结构发生重大变化

### 5.2 维护步骤
1. 从同花顺 iFinD 或基金官网导出最新 ETF 清单
2. 按初筛标准（规模 >= 2亿、跟踪误差 <= 5%、日均成交 >= 5000万、费率 <= 0.6%）筛选
3. 更新 `etf_universe.csv` 中的基础字段（code, name, fund_size, tracking_error, daily_turnover, fee 等）
4. 手工核对并修正 `market_cap_label` 和 `pool_role`
5. 同步更新本文档中的字段说明（如有新增取值）

### 5.3 注意事项
- **不要随意删除历史记录**：即使某只 ETF 被降级到 `extended` 或淘汰，也建议保留行记录并将其 `pool_role` 改为 `exclude`，以便回溯研究
- **标签冲突处理**：当一只 ETF 可以归入多个 `market_cap_label` 时，优先选择**最能代表其投资特征的单一标签**（互斥原则）
- **缺失数据标注**：对于无法获取的数据，统一留空，不要用 `0` 或 `--` 填充

## 6. 常见问题

**Q1：为什么货币基金（如 511880.SH）的 `fund_size` 为空，但 `daily_turnover` 很大？**
A：原始数据中货币基金的"基金规模"列未填充，但"年日均成交量"反映了其场内交易活跃度。对于货基，建议优先参考 `daily_turnover` 判断流动性。

**Q2：`market_cap_label` 的取值不准确怎么办？**
A：v2 中的 `market_cap_label` 部分基于规则自动推断、部分继承自原数据。**必须经过人工审核**。你可以在 Excel 或 CSV 编辑器中直接修改该列，然后保存。

**Q3：核心池的数量是否有硬性上限？**
A：当前 v2 的核心池约为 14 只，但这不是硬性约束。用户可以根据组合构建需求调整核心池大小，一般建议控制在 **10-18 只** 以内，以保证组合的可管理性。

**Q4：行业主题 ETF 的 `market_cap_label` 为什么大多是空的？**
A：行业主题 ETF 的首要分类维度是 `sector_l1` / `sector_l2`，市值属性相对次要。如果未来需要为行业主题 ETF 补充市值/市场定位标签，可以在此字段填充（如 `科技大盘` / `医药中盘` / `港股医药`）。
